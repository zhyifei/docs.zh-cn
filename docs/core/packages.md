# <a name="packages-metapackages-and-frameworks"></a><span data-ttu-id="b5164-104">包、元包和框架</span><span class="sxs-lookup"><span data-stu-id="b5164-104">Packages, Metapackages and Frameworks</span></span>

<span data-ttu-id="b5164-105">.NET Core 是一种由 NuGet 包组成的平台。</span><span class="sxs-lookup"><span data-stu-id="b5164-105">.NET Core is a platform made of NuGet packages.</span></span> <span data-ttu-id="b5164-106">有些产品体验受益于包的细粒度定义，而另一些受益于粗粒度的定义。</span><span class="sxs-lookup"><span data-stu-id="b5164-106">Some product experiences benefit from fine-grained definition of packages while others from coarse-grained.</span></span> <span data-ttu-id="b5164-107">为了适应这种二元定义，一款好的产品应该作为一组细粒度的包发布，然后在更粗的粒度组块中进行描述，单个包的正式的名字叫做“元包”。</span><span class="sxs-lookup"><span data-stu-id="b5164-107">To accommodate this duality, the product is distributed as a fine-grained set of packages and then described in coarser chunks with a package type informally called a "metapackage".</span></span>

<span data-ttu-id="b5164-108">每个 .Net Core 包都支持以框架形式通过多个 .Net 实现代码运行。</span><span class="sxs-lookup"><span data-stu-id="b5164-108">Each of the .NET Core packages support being run on multiple .NET implementations, represented as frameworks.</span></span> <span data-ttu-id="b5164-109">其中有些框架是传统框架，例如表示 .NET Framework 的 `net46`。</span><span class="sxs-lookup"><span data-stu-id="b5164-109">Some of those frameworks are traditional frameworks, like `net46`, representing the .NET Framework.</span></span> <span data-ttu-id="b5164-110">而另一些则是新框架，可视为是“基于包的框架”，这种是框架的另外一种新的定义模型。</span><span class="sxs-lookup"><span data-stu-id="b5164-110">Another set is new frameworks that can be thought of as "package-based frameworks", which establish a new model for defining frameworks.</span></span> <span data-ttu-id="b5164-111">这些基于包的框架整个都是由包组成的，它们自身也被定义成包，这就在包与框架之间形成了一种比较密切的关系。</span><span class="sxs-lookup"><span data-stu-id="b5164-111">These package-based frameworks are entirely formed and defined as packages, forming a strong relationship between packages and frameworks.</span></span>

## <a name="packages"></a><span data-ttu-id="b5164-112">包</span><span class="sxs-lookup"><span data-stu-id="b5164-112">Packages</span></span>

<span data-ttu-id="b5164-113">.NET Core 被分成一组包，它们提供了基元类型，以及更高层的数据类型，应用组合类型和通用实用工具。</span><span class="sxs-lookup"><span data-stu-id="b5164-113">.NET Core is split into a set of packages, which provide primitives, higher-level data types, app composition types and common utilities.</span></span> <span data-ttu-id="b5164-114">每一个包都代表着单独的同名程序集。</span><span class="sxs-lookup"><span data-stu-id="b5164-114">Each of these packages represent a single assembly of the same name.</span></span> <span data-ttu-id="b5164-115">例如，[System.Runtime](https://www.nuget.org/packages/System.Runtime) 这个包就包含了 System.Runtime.dll 程序集。</span><span class="sxs-lookup"><span data-stu-id="b5164-115">For example, [System.Runtime](https://www.nuget.org/packages/System.Runtime) contains System.Runtime.dll.</span></span> 

<span data-ttu-id="b5164-116">以细粒度方式定义这些包具有以下好处：</span><span class="sxs-lookup"><span data-stu-id="b5164-116">There are advantages to defining packages in a fine-grained manner:</span></span>

- <span data-ttu-id="b5164-119">细粒度的包可以单独依赖于某一个库。</span><span class="sxs-lookup"><span data-stu-id="b5164-119">Fine-grained packages can have dependencies specific to only one library.</span></span>

<span data-ttu-id="b5164-121">上述某些好处只适用于某些特定场合。</span><span class="sxs-lookup"><span data-stu-id="b5164-121">Some of these benefits are only used in certain circumstances.</span></span> <span data-ttu-id="b5164-122">例如，.NET Core 的所有包通常都会在同一计划内提供对同一平台的支持。</span><span class="sxs-lookup"><span data-stu-id="b5164-122">For example, NET Core packages will typically ship on the same schedule with the same platform support.</span></span> <span data-ttu-id="b5164-123">在这种情况下，补丁与更新会以小的单独包的形式发布和安装。</span><span class="sxs-lookup"><span data-stu-id="b5164-123">In the case of servicing, fixes can be distributed and installed as small single package updates.</span></span> <span data-ttu-id="b5164-124">由于这种小范围的变化，补丁的验证与时间花费，都可以限制到单个库的需求范围中。</span><span class="sxs-lookup"><span data-stu-id="b5164-124">Due to the narrow scope of change, the validation and time to make a fix available is limited to what is needed for a single library.</span></span>
