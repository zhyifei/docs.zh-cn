### <a name="memory-allocation"></a><span data-ttu-id="3a588-143">内存分配</span><span class="sxs-lookup"><span data-stu-id="3a588-143">Memory allocation</span></span>

<span data-ttu-id="3a588-144">初始化新进程时，运行时会为进程保留一个连续的地址空间区域。</span><span class="sxs-lookup"><span data-stu-id="3a588-144">When you initialize a new process, the runtime reserves a contiguous region of address space for the process.</span></span> <span data-ttu-id="3a588-145">这个保留的地址空间被称为托管堆。</span><span class="sxs-lookup"><span data-stu-id="3a588-145">This reserved address space is called the managed heap.</span></span> <span data-ttu-id="3a588-146">托管堆维护着一个指针，用它指向将在堆中分配的下一个对象的地址。</span><span class="sxs-lookup"><span data-stu-id="3a588-146">The managed heap maintains a pointer to the address where the next object in the heap will be allocated.</span></span> <span data-ttu-id="3a588-147">最初，该指针设置为指向托管堆的基址。</span><span class="sxs-lookup"><span data-stu-id="3a588-147">Initially, this pointer is set to the managed heap's base address.</span></span> <span data-ttu-id="3a588-148">托管堆上部署了所有引用类型。</span><span class="sxs-lookup"><span data-stu-id="3a588-148">All reference types are allocated on the managed heap.</span></span> <span data-ttu-id="3a588-149">应用程序创建第一个引用类型时，将为托管堆的基址中的类型分配内存。</span><span class="sxs-lookup"><span data-stu-id="3a588-149">When an application creates the first reference type, memory is allocated for the type at the base address of the managed heap.</span></span> <span data-ttu-id="3a588-150">应用程序创建下一个对象时，垃圾回收器在紧接第一个对象后面的地址空间内为它分配内存。</span><span class="sxs-lookup"><span data-stu-id="3a588-150">When the application creates the next object, the garbage collector allocates memory for it in the address space immediately following the first object.</span></span> <span data-ttu-id="3a588-151">只要地址空间可用，垃圾回收器就会继续以这种方式为新对象分配空间。</span><span class="sxs-lookup"><span data-stu-id="3a588-151">As long as address space is available, the garbage collector continues to allocate space for new objects in this manner.</span></span>

<span data-ttu-id="3a588-152">从托管堆中分配内存要比非托管内存分配速度快。</span><span class="sxs-lookup"><span data-stu-id="3a588-152">Allocating memory from the managed heap is faster than unmanaged memory allocation.</span></span> <span data-ttu-id="3a588-153">由于运行时通过为指针添加值来为对象分配内存，所以这几乎和从堆栈中分配内存一样快。</span><span class="sxs-lookup"><span data-stu-id="3a588-153">Because the runtime allocates memory for an object by adding a value to a pointer, it's almost as fast as allocating memory from the stack.</span></span> <span data-ttu-id="3a588-154">另外，由于连续分配的新对象在托管堆中是连续存储，所以应用程序可以快速访问这些对象。</span><span class="sxs-lookup"><span data-stu-id="3a588-154">In addition, because new objects that are allocated consecutively are stored contiguously in the managed heap, an application can access the objects quickly.</span></span>

### <a name="memory-release"></a><span data-ttu-id="3a588-155">内存释放</span><span class="sxs-lookup"><span data-stu-id="3a588-155">Memory release</span></span>

<span data-ttu-id="3a588-156">垃圾回收器的优化引擎根据所执行的分配决定执行回收的最佳时间。</span><span class="sxs-lookup"><span data-stu-id="3a588-156">The garbage collector's optimizing engine determines the best time to perform a collection based on the allocations being made.</span></span> <span data-ttu-id="3a588-157">垃圾回收器在执行回收时，会释放应用程序不再使用的对象的内存。</span><span class="sxs-lookup"><span data-stu-id="3a588-157">When the garbage collector performs a collection, it releases the memory for objects that are no longer being used by the application.</span></span> <span data-ttu-id="3a588-158">它通过检查应用程序的根  来确定不再使用的对象。</span><span class="sxs-lookup"><span data-stu-id="3a588-158">It determines which objects are no longer being used by examining the application's *roots*.</span></span> <span data-ttu-id="3a588-159">应用程序的根包含线程堆栈上的静态字段、局部变量和参数以及 CPU 寄存器。</span><span class="sxs-lookup"><span data-stu-id="3a588-159">An application's roots include static fields, local variables and parameters on a thread's stack, and CPU registers.</span></span> <span data-ttu-id="3a588-160">每个根或者引用托管堆中的对象，或者设置为空。</span><span class="sxs-lookup"><span data-stu-id="3a588-160">Each root either refers to an object on the managed heap or is set to null.</span></span> <span data-ttu-id="3a588-161">垃圾回收器可以访问由实时 (JIT) 编译器和运行时维护的活动根的列表。</span><span class="sxs-lookup"><span data-stu-id="3a588-161">The garbage collector has access to the list of active roots that the just-in-time (JIT) compiler and the runtime maintain.</span></span> <span data-ttu-id="3a588-162">垃圾回收器使用此列表创建一个图表，其中包含所有可从这些根中访问的对象。</span><span class="sxs-lookup"><span data-stu-id="3a588-162">Using this list, the garbage collector creates a graph that contains all the objects that are reachable from the roots.</span></span>

<span data-ttu-id="3a588-163">不在该图表中的对象将无法从应用程序的根中访问。</span><span class="sxs-lookup"><span data-stu-id="3a588-163">Objects that are not in the graph are unreachable from the application's roots.</span></span> <span data-ttu-id="3a588-164">垃圾回收器会考虑无法访问的对象垃圾，并释放为它们分配的内存。</span><span class="sxs-lookup"><span data-stu-id="3a588-164">The garbage collector considers unreachable objects garbage and releases the memory allocated for them.</span></span> <span data-ttu-id="3a588-165">在回收中，垃圾回收器检查托管堆，查找无法访问对象所占据的地址空间块。</span><span class="sxs-lookup"><span data-stu-id="3a588-165">During a collection, the garbage collector examines the managed heap, looking for the blocks of address space occupied by unreachable objects.</span></span> <span data-ttu-id="3a588-166">发现无法访问的对象时，它就使用内存复制功能来压缩内存中可以访问的对象，释放分配给不可访问对象的地址空间块。</span><span class="sxs-lookup"><span data-stu-id="3a588-166">As it discovers each unreachable object, it uses a memory-copying function to compact the reachable objects in memory, freeing up the blocks of address spaces allocated to unreachable objects.</span></span> <span data-ttu-id="3a588-167">在压缩了可访问对象的内存后，垃圾回收器就会做出必要的指针更正，以便应用程序的根指向新地址中的对象。</span><span class="sxs-lookup"><span data-stu-id="3a588-167">Once the memory for the reachable objects has been compacted, the garbage collector makes the necessary pointer corrections so that the application's roots point to the objects in their new locations.</span></span> <span data-ttu-id="3a588-168">它还将托管堆指针定位至最后一个可访问对象之后。</span><span class="sxs-lookup"><span data-stu-id="3a588-168">It also positions the managed heap's pointer after the last reachable object.</span></span>

## <a name="the-managed-heap"></a><span data-ttu-id="3a588-183">托管堆</span><span class="sxs-lookup"><span data-stu-id="3a588-183">The managed heap</span></span>

<span data-ttu-id="3a588-184">在垃圾回收器由 CLR 初始化之后，它会分配一段内存用于存储和管理对象。</span><span class="sxs-lookup"><span data-stu-id="3a588-184">After the garbage collector is initialized by the CLR, it allocates a segment of memory to store and manage objects.</span></span> <span data-ttu-id="3a588-185">此内存称为托管堆（与操作系统中的本机堆相对）。</span><span class="sxs-lookup"><span data-stu-id="3a588-185">This memory is called the managed heap, as opposed to a native heap in the operating system.</span></span>

<span data-ttu-id="3a588-186">每个托管进程都有一个托管堆。</span><span class="sxs-lookup"><span data-stu-id="3a588-186">There is a managed heap for each managed process.</span></span> <span data-ttu-id="3a588-187">进程中的所有线程都在同一堆上为对象分配内存。</span><span class="sxs-lookup"><span data-stu-id="3a588-187">All threads in the process allocate memory for objects on the same heap.</span></span>

<span data-ttu-id="3a588-188">若要保留内存，垃圾回收器会调用 Windows [VirtualAlloc](/windows/desktop/api/memoryapi/nf-memoryapi-virtualalloc) 函数，并且每次为托管应用保留一个内存段。</span><span class="sxs-lookup"><span data-stu-id="3a588-188">To reserve memory, the garbage collector calls the Windows [VirtualAlloc](/windows/desktop/api/memoryapi/nf-memoryapi-virtualalloc) function and reserves one segment of memory at a time for managed applications.</span></span> <span data-ttu-id="3a588-189">垃圾回收器还会根据需要保留内存段，并调用 Windows [VirtualFree](/windows/desktop/api/memoryapi/nf-memoryapi-virtualfree) 函数，将内存段释放回操作系统（在清除所有对象的内存段后）。</span><span class="sxs-lookup"><span data-stu-id="3a588-189">The garbage collector also reserves segments, as needed, and releases segments back to the operating system (after clearing them of any objects) by calling the Windows [VirtualFree](/windows/desktop/api/memoryapi/nf-memoryapi-virtualfree) function.</span></span>

> [!IMPORTANT]
> <span data-ttu-id="3a588-190">垃圾回收器分配的段大小特定于实现，并且随时可能更改（包括定期更新）。</span><span class="sxs-lookup"><span data-stu-id="3a588-190">The size of segments allocated by the garbage collector is implementation-specific and is subject to change at any time, including in periodic updates.</span></span> <span data-ttu-id="3a588-191">应用程序不应假设特定段的大小或依赖于此大小，也不应尝试配置段分配可用的内存量。</span><span class="sxs-lookup"><span data-stu-id="3a588-191">Your app should never make assumptions about or depend on a particular segment size, nor should it attempt to configure the amount of memory available for segment allocations.</span></span>

<span data-ttu-id="3a588-192">堆上分配的对象越少，垃圾回收器必须执行的工作就越少。</span><span class="sxs-lookup"><span data-stu-id="3a588-192">The fewer objects allocated on the heap, the less work the garbage collector has to do.</span></span> <span data-ttu-id="3a588-193">分配对象时，请勿使用超出你需求的舍入值，例如在仅需要 15 个字节的情况下分配了 32 个字节的数组。</span><span class="sxs-lookup"><span data-stu-id="3a588-193">When you allocate objects, don't use rounded-up values that exceed your needs, such as allocating an array of 32 bytes when you need only 15 bytes.</span></span>

<span data-ttu-id="3a588-194">当触发垃圾回收时，垃圾回收器将回收由非活动对象占用的内存。</span><span class="sxs-lookup"><span data-stu-id="3a588-194">When a garbage collection is triggered, the garbage collector reclaims the memory that's occupied by dead objects.</span></span> <span data-ttu-id="3a588-195">回收进程会对活动对象进行压缩，以便将它们一起移动，并移除死空间，从而使堆更小一些。</span><span class="sxs-lookup"><span data-stu-id="3a588-195">The reclaiming process compacts live objects so that they are moved together, and the dead space is removed, thereby making the heap smaller.</span></span> <span data-ttu-id="3a588-196">这将确保一起分配的对象全都位于托管堆上，从而保留它们的局部性。</span><span class="sxs-lookup"><span data-stu-id="3a588-196">This ensures that objects that are allocated together stay together on the managed heap to preserve their locality.</span></span>

<span data-ttu-id="3a588-197">垃圾回收的侵入性（频率和持续时间）是由分配的数量和托管堆上保留的内存数量决定的。</span><span class="sxs-lookup"><span data-stu-id="3a588-197">The intrusiveness (frequency and duration) of garbage collections is the result of the volume of allocations and the amount of survived memory on the managed heap.</span></span>

<span data-ttu-id="3a588-198">此堆可视为两个堆的累计：[大对象堆](large-object-heap.md)和小对象堆。</span><span class="sxs-lookup"><span data-stu-id="3a588-198">The heap can be considered as the accumulation of two heaps: the [large object heap](large-object-heap.md) and the small object heap.</span></span> <span data-ttu-id="3a588-199">大对象堆包含大小为 85,000 个字节和更多字节的对象，这些对象通常是数组。</span><span class="sxs-lookup"><span data-stu-id="3a588-199">The large object heap contains objects that are 85,000 bytes and larger, which are usually arrays.</span></span> <span data-ttu-id="3a588-200">非常大的实例对象是很少见的。</span><span class="sxs-lookup"><span data-stu-id="3a588-200">It's rare for an instance object to be extremely large.</span></span>

> [!TIP]
> <span data-ttu-id="3a588-201">可以[配置阈值大小](../../core/run-time-config/garbage-collector.md#large-object-heap-threshold)，以使对象能够进入大型对象堆。</span><span class="sxs-lookup"><span data-stu-id="3a588-201">You can [configure the threshold size](../../core/run-time-config/garbage-collector.md#large-object-heap-threshold) for objects to go on the large object heap.</span></span>

## <a name="generations"></a><span data-ttu-id="3a588-202">代数</span><span class="sxs-lookup"><span data-stu-id="3a588-202">Generations</span></span>

<span data-ttu-id="3a588-203">GC 算法基于几个注意事项：</span><span class="sxs-lookup"><span data-stu-id="3a588-203">The GC algorithm is based on several considerations:</span></span>

- <span data-ttu-id="3a588-204">压缩托管堆的一部分内存要比压缩整个托管堆速度快。</span><span class="sxs-lookup"><span data-stu-id="3a588-204">It's faster to compact the memory for a portion of the managed heap than for the entire managed heap.</span></span>
- <span data-ttu-id="3a588-205">较新的对象生存期较短，而较旧的对象生存期则较长。</span><span class="sxs-lookup"><span data-stu-id="3a588-205">Newer objects have shorter lifetimes and older objects have longer lifetimes.</span></span>
- <span data-ttu-id="3a588-206">较新的对象趋向于相互关联，并且大致同时由应用程序访问。</span><span class="sxs-lookup"><span data-stu-id="3a588-206">Newer objects tend to be related to each other and accessed by the application around the same time.</span></span>

<span data-ttu-id="3a588-207">垃圾回收主要在回收短生存期对象时发生。</span><span class="sxs-lookup"><span data-stu-id="3a588-207">Garbage collection primarily occurs with the reclamation of short-lived objects.</span></span> <span data-ttu-id="3a588-208">为优化垃圾回收器的性能，将托管堆分为三代：第 0 代、第 1 代和第 2 代，因此它可以单独处理长生存期和短生存期对象。</span><span class="sxs-lookup"><span data-stu-id="3a588-208">To optimize the performance of the garbage collector, the managed heap is divided into three generations, 0, 1, and 2, so it can handle long-lived and short-lived objects separately.</span></span> <span data-ttu-id="3a588-209">垃圾回收器将新对象存储在第 0 代中。</span><span class="sxs-lookup"><span data-stu-id="3a588-209">The garbage collector stores new objects in generation 0.</span></span> <span data-ttu-id="3a588-210">在应用程序生存期的早期创建的对象如果未被回收，则被升级并存储在第 1 级和第 2 级中。</span><span class="sxs-lookup"><span data-stu-id="3a588-210">Objects created early in the application's lifetime that survive collections are promoted and stored in generations 1 and 2.</span></span> <span data-ttu-id="3a588-211">因为压缩托管堆的一部分要比压缩整个托管堆速度快，所以此方案允许垃圾回收器在每次执行回收时释放特定级别的内存，而不是整个托管堆的内存。</span><span class="sxs-lookup"><span data-stu-id="3a588-211">Because it's faster to compact a portion of the managed heap than the entire heap, this scheme allows the garbage collector to release the memory in a specific generation rather than release the memory for the entire managed heap each time it performs a collection.</span></span>

- <span data-ttu-id="3a588-212">**第 0 代**。</span><span class="sxs-lookup"><span data-stu-id="3a588-212">**Generation 0**.</span></span> <span data-ttu-id="3a588-213">这是最年轻的代，其中包含短生存期对象。</span><span class="sxs-lookup"><span data-stu-id="3a588-213">This is the youngest generation and contains short-lived objects.</span></span> <span data-ttu-id="3a588-214">短生存期对象的一个示例是临时变量。</span><span class="sxs-lookup"><span data-stu-id="3a588-214">An example of a short-lived object is a temporary variable.</span></span> <span data-ttu-id="3a588-215">垃圾回收最常发生在此代中。</span><span class="sxs-lookup"><span data-stu-id="3a588-215">Garbage collection occurs most frequently in this generation.</span></span>

  <span data-ttu-id="3a588-216">新分配的对象构成新一代对象，并隐式地成为第 0 代集合。</span><span class="sxs-lookup"><span data-stu-id="3a588-216">Newly allocated objects form a new generation of objects and are implicitly generation 0 collections.</span></span> <span data-ttu-id="3a588-217">但是，如果它们是大型对象，它们将延续到第 2 代集合中的大型对象堆。</span><span class="sxs-lookup"><span data-stu-id="3a588-217">However, if they are large objects, they go on the large object heap in a generation 2 collection.</span></span>

  <span data-ttu-id="3a588-218">大多数对象通过第 0 代中的垃圾回收进行回收，不会保留到下一代。</span><span class="sxs-lookup"><span data-stu-id="3a588-218">Most objects are reclaimed for garbage collection in generation 0 and don't survive to the next generation.</span></span>
  
  <span data-ttu-id="3a588-219">如果应用程序在第 0 代托管堆已满时尝试创建新对象，垃圾回收器将执行收集，以尝试为该对象释放地址空间。</span><span class="sxs-lookup"><span data-stu-id="3a588-219">If an application attempts to create a new object when generation 0 is full, the garbage collector performs a collection in an attempt to free address space for the object.</span></span> <span data-ttu-id="3a588-220">垃圾回收器从检查第 0 级托管堆中的对象（而不是托管堆中的所有对象）开始执行回收。</span><span class="sxs-lookup"><span data-stu-id="3a588-220">The garbage collector starts by examining the objects in generation 0 rather than all objects in the managed heap.</span></span> <span data-ttu-id="3a588-221">单独回收第 0 代托管堆通常可以回收足够的内存，这样，应用程序便可以继续创建新对象。</span><span class="sxs-lookup"><span data-stu-id="3a588-221">A collection of generation 0 alone often reclaims enough memory to enable the application to continue creating new objects.</span></span>

- <span data-ttu-id="3a588-222">**第 1 代**。</span><span class="sxs-lookup"><span data-stu-id="3a588-222">**Generation 1**.</span></span> <span data-ttu-id="3a588-223">这一代包含短生存期对象并用作短生存期对象和长生存期对象之间的缓冲区。</span><span class="sxs-lookup"><span data-stu-id="3a588-223">This generation contains short-lived objects and serves as a buffer between short-lived objects and long-lived objects.</span></span>

  <span data-ttu-id="3a588-224">垃圾回收器执行第 0 代托管堆的回收后，会压缩可访问对象的内存，并将其升级到第 1 代。</span><span class="sxs-lookup"><span data-stu-id="3a588-224">After the garbage collector performs a collection of generation 0, it compacts the memory for the reachable objects and promotes them to generation 1.</span></span> <span data-ttu-id="3a588-225">因为未被回收的对象往往具有较长的生存期，所以将它们升级至更高的级别很有意义。</span><span class="sxs-lookup"><span data-stu-id="3a588-225">Because objects that survive collections tend to have longer lifetimes, it makes sense to promote them to a higher generation.</span></span> <span data-ttu-id="3a588-226">垃圾回收器在每次执行第 0 代托管堆的回收时，不必重新检查第 1 代和第 2 代托管堆中的对象。</span><span class="sxs-lookup"><span data-stu-id="3a588-226">The garbage collector doesn't have to reexamine the objects in generations 1 and 2 each time it performs a collection of generation 0.</span></span>
  
  <span data-ttu-id="3a588-227">如果第 0 代托管堆的回收没有回收足够的内存供应用程序创建新对象，垃圾回收器就会先执行第 1 代托管堆的回收，然后再执行第 2 代托管堆的回收。</span><span class="sxs-lookup"><span data-stu-id="3a588-227">If a collection of generation 0 does not reclaim enough memory for the application to create a new object, the garbage collector can perform a collection of generation 1, then generation 2.</span></span> <span data-ttu-id="3a588-228">第 1 级托管堆中未被回收的对象将会升级至第 2 级托管堆。</span><span class="sxs-lookup"><span data-stu-id="3a588-228">Objects in generation 1 that survive collections are promoted to generation 2.</span></span>

- <span data-ttu-id="3a588-229">**第 2 代**。</span><span class="sxs-lookup"><span data-stu-id="3a588-229">**Generation 2**.</span></span> <span data-ttu-id="3a588-230">这一代包含长生存期对象。</span><span class="sxs-lookup"><span data-stu-id="3a588-230">This generation contains long-lived objects.</span></span> <span data-ttu-id="3a588-231">长生存期对象的一个示例是服务器应用程序中的一个包含在进程期间处于活动状态的静态数据的对象。</span><span class="sxs-lookup"><span data-stu-id="3a588-231">An example of a long-lived object is an object in a server application that contains static data that's live for the duration of the process.</span></span>

  <span data-ttu-id="3a588-232">第 2 代托管堆中未被回收的对象会继续保留在第 2 代托管堆中，直到在将来的回收中确定它们无法访问为止。</span><span class="sxs-lookup"><span data-stu-id="3a588-232">Objects in generation 2 that survive a collection remain in generation 2 until they are determined to be unreachable in a future collection.</span></span>

<span data-ttu-id="3a588-233">当条件得到满足时，垃圾回收将在特定代上发生。</span><span class="sxs-lookup"><span data-stu-id="3a588-233">Garbage collections occur on specific generations as conditions warrant.</span></span> <span data-ttu-id="3a588-234">回收某个代意味着回收此代中的对象及其所有更年轻的代。</span><span class="sxs-lookup"><span data-stu-id="3a588-234">Collecting a generation means collecting objects in that generation and all its younger generations.</span></span> <span data-ttu-id="3a588-235">第 2 代垃圾回收也称为完整垃圾回收，因为它回收所有代中的对象（即，托管堆中的所有对象）。</span><span class="sxs-lookup"><span data-stu-id="3a588-235">A generation 2 garbage collection is also known as a full garbage collection, because it reclaims objects in all generations (that is, all objects in the managed heap).</span></span>

### <a name="survival-and-promotions"></a><span data-ttu-id="3a588-236">幸存和提升</span><span class="sxs-lookup"><span data-stu-id="3a588-236">Survival and promotions</span></span>

<span data-ttu-id="3a588-237">垃圾回收中未回收的对象也称为幸存者，并会被提升到下一代：</span><span class="sxs-lookup"><span data-stu-id="3a588-237">Objects that are not reclaimed in a garbage collection are known as survivors and are promoted to the next generation:</span></span>

- <span data-ttu-id="3a588-238">第 0 代垃圾回收中未被回收的对象将会升级至第 1 代。</span><span class="sxs-lookup"><span data-stu-id="3a588-238">Objects that survive a generation 0 garbage collection are promoted to generation 1.</span></span>
- <span data-ttu-id="3a588-239">第 1 代垃圾回收中未被回收的对象将会升级至第 2 代。</span><span class="sxs-lookup"><span data-stu-id="3a588-239">Objects that survive a generation 1 garbage collection are promoted to generation 2.</span></span>
- <span data-ttu-id="3a588-240">第 2 代垃圾回收中未被回收的对象将仍保留在第 2 代。</span><span class="sxs-lookup"><span data-stu-id="3a588-240">Objects that survive a generation 2 garbage collection remain in generation 2.</span></span>

<span data-ttu-id="3a588-241">当垃圾回收器检测到某个代中的幸存率很高时，它会增加该代的分配阈值。</span><span class="sxs-lookup"><span data-stu-id="3a588-241">When the garbage collector detects that the survival rate is high in a generation, it increases the threshold of allocations for that generation.</span></span> <span data-ttu-id="3a588-242">下次回收将回收非常大的内存。</span><span class="sxs-lookup"><span data-stu-id="3a588-242">The next collection gets a substantial size of reclaimed memory.</span></span> <span data-ttu-id="3a588-243">CLR 持续在以下两个优先级之间进行平衡：不允许通过延迟垃圾回收，让应用程序的工作集获取太大内存，以及不允许垃圾回收过于频繁地运行。</span><span class="sxs-lookup"><span data-stu-id="3a588-243">The CLR continually balances two priorities: not letting an application's working set get too large by delaying garbage collection and not letting the garbage collection run too frequently.</span></span>

## <a name="what-happens-during-a-garbage-collection"></a><span data-ttu-id="3a588-270">垃圾回收过程中发生的情况</span><span class="sxs-lookup"><span data-stu-id="3a588-270">What happens during a garbage collection</span></span>

<span data-ttu-id="3a588-271">垃圾回收分为以下几个阶段：</span><span class="sxs-lookup"><span data-stu-id="3a588-271">A garbage collection has the following phases:</span></span>

- <span data-ttu-id="3a588-272">标记阶段，找到并创建所有活动对象的列表。</span><span class="sxs-lookup"><span data-stu-id="3a588-272">A marking phase that finds and creates a list of all live objects.</span></span>

- <span data-ttu-id="3a588-273">重定位阶段，用于更新对将要压缩的对象的引用。</span><span class="sxs-lookup"><span data-stu-id="3a588-273">A relocating phase that updates the references to the objects that will be compacted.</span></span>

- <span data-ttu-id="3a588-274">压缩阶段，用于回收由死对象占用的空间，并压缩幸存的对象。</span><span class="sxs-lookup"><span data-stu-id="3a588-274">A compacting phase that reclaims the space occupied by the dead objects and compacts the surviving objects.</span></span> <span data-ttu-id="3a588-275">压缩阶段将垃圾回收中幸存下来的对象移至段中时间较早的一端。</span><span class="sxs-lookup"><span data-stu-id="3a588-275">The compacting phase moves objects that have survived a garbage collection toward the older end of the segment.</span></span>

  <span data-ttu-id="3a588-276">因为第 2 代回收可以占用多个段，所以可以将已提升到第 2 代中的对象移动到时间较早的段中。</span><span class="sxs-lookup"><span data-stu-id="3a588-276">Because generation 2 collections can occupy multiple segments, objects that are promoted into generation 2 can be moved into an older segment.</span></span> <span data-ttu-id="3a588-277">可以将第 1 代幸存者和第 2 代幸存者都移动到不同的段，因为它们已被提升到第 2 代。</span><span class="sxs-lookup"><span data-stu-id="3a588-277">Both generation 1 and generation 2 survivors can be moved to a different segment, because they are promoted to generation 2.</span></span>

  <span data-ttu-id="3a588-278">通常，由于复制大型对象会造成性能代偿，因此不会压缩大型对象堆 (LOH)。</span><span class="sxs-lookup"><span data-stu-id="3a588-278">Ordinarily, the large object heap (LOH) is not compacted, because copying large objects imposes a performance penalty.</span></span> <span data-ttu-id="3a588-279">但是，在 .NET Core 和 .NET Framework 4.5.1 及更高版本中，可以根据需要使用 <xref:System.Runtime.GCSettings.LargeObjectHeapCompactionMode%2A?displayProperty=nameWithType> 属性按需压缩大型对象堆。</span><span class="sxs-lookup"><span data-stu-id="3a588-279">However, in .NET Core and in .NET Framework 4.5.1 and later, you can use the <xref:System.Runtime.GCSettings.LargeObjectHeapCompactionMode%2A?displayProperty=nameWithType> property to compact the large object heap on demand.</span></span> <span data-ttu-id="3a588-280">此外，当通过指定以下任一项设置硬限制时，将自动压缩 LOH：</span><span class="sxs-lookup"><span data-stu-id="3a588-280">In addition, the LOH is automatically compacted when a hard limit is set by specifying either:</span></span>

  - <span data-ttu-id="3a588-281">针对容器的内存限制。</span><span class="sxs-lookup"><span data-stu-id="3a588-281">A memory limit on a container.</span></span>
  - <span data-ttu-id="3a588-282">[GCHeapHardLimit](../../core/run-time-config/garbage-collector.md#systemgcheaphardlimitcomplus_gcheaphardlimit) 或 [GCHeapHardLimitPercent](../../core/run-time-config/garbage-collector.md#systemgcheaphardlimitpercentcomplus_gcheaphardlimitpercent) 运行时配置选项。</span><span class="sxs-lookup"><span data-stu-id="3a588-282">The [GCHeapHardLimit](../../core/run-time-config/garbage-collector.md#systemgcheaphardlimitcomplus_gcheaphardlimit) or [GCHeapHardLimitPercent](../../core/run-time-config/garbage-collector.md#systemgcheaphardlimitpercentcomplus_gcheaphardlimitpercent) run-time configuration options.</span></span>

<span data-ttu-id="3a588-283">垃圾回收器使用以下信息来确定对象是否为活动对象：</span><span class="sxs-lookup"><span data-stu-id="3a588-283">The garbage collector uses the following information to determine whether objects are live:</span></span>

- <span data-ttu-id="3a588-284">**堆栈根**。</span><span class="sxs-lookup"><span data-stu-id="3a588-284">**Stack roots**.</span></span> <span data-ttu-id="3a588-285">由实时 (JIT) 编译器和堆栈查看器提供的堆栈变量。</span><span class="sxs-lookup"><span data-stu-id="3a588-285">Stack variables provided by the just-in-time (JIT) compiler and stack walker.</span></span> <span data-ttu-id="3a588-286">JIT 优化可以延长或缩短报告给垃圾回收器的堆栈变量内的代码的区域。</span><span class="sxs-lookup"><span data-stu-id="3a588-286">JIT optimizations can lengthen or shorten regions of code within which stack variables are reported to the garbage collector.</span></span>

- <span data-ttu-id="3a588-287">**垃圾回收句柄**。</span><span class="sxs-lookup"><span data-stu-id="3a588-287">**Garbage collection handles**.</span></span> <span data-ttu-id="3a588-288">指向托管对象且可由用户代码或公共语言运行时分配的句柄。</span><span class="sxs-lookup"><span data-stu-id="3a588-288">Handles that point to managed objects and that can be allocated by user code or by the common language runtime.</span></span>

- <span data-ttu-id="3a588-289">**静态数据**。</span><span class="sxs-lookup"><span data-stu-id="3a588-289">**Static data**.</span></span> <span data-ttu-id="3a588-290">应用程序域中可能引用其他对象的静态对象。</span><span class="sxs-lookup"><span data-stu-id="3a588-290">Static objects in application domains that could be referencing other objects.</span></span> <span data-ttu-id="3a588-291">每个应用程序域都会跟踪其静态对象。</span><span class="sxs-lookup"><span data-stu-id="3a588-291">Each application domain keeps track of its static objects.</span></span>
