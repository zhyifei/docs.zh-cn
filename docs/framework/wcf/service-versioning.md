## <a name="contract-versioning"></a><span data-ttu-id="3a7cd-121">协定版本管理</span><span class="sxs-lookup"><span data-stu-id="3a7cd-121">Contract Versioning</span></span>

 <span data-ttu-id="3a7cd-124">对于数据协定，兼容性意味着可以添加新的架构类型定义，但是不能以中断性方式更改架构类型定义。</span><span class="sxs-lookup"><span data-stu-id="3a7cd-124">For data contracts, compatibility means new schema type definitions can be added but existing schema type definitions cannot be changed in breaking ways.</span></span> <span data-ttu-id="3a7cd-125">中断性更改可包括移除数据成员或不兼容地更改其数据类型。</span><span class="sxs-lookup"><span data-stu-id="3a7cd-125">Breaking changes might include removing data members or changing their data type incompatibly.</span></span> <span data-ttu-id="3a7cd-126">这一特点允许服务在某种程度上更改其协定的版本而又不中断客户端。</span><span class="sxs-lookup"><span data-stu-id="3a7cd-126">This feature allows the service some latitude in changing the version of its contracts without breaking clients.</span></span> <span data-ttu-id="3a7cd-127">接下来的两部分介绍非中断性和重大更改，可以对 WCF 数据服务协定。</span><span class="sxs-lookup"><span data-stu-id="3a7cd-127">The next two sections explain nonbreaking and breaking changes that can be made to WCF data and service contracts.</span></span>  
  
## <a name="data-contract-versioning"></a><span data-ttu-id="3a7cd-128">数据协定版本管理</span><span class="sxs-lookup"><span data-stu-id="3a7cd-128">Data Contract Versioning</span></span>

### <a name="strict-versioning"></a><span data-ttu-id="3a7cd-130">严格版本管理</span><span class="sxs-lookup"><span data-stu-id="3a7cd-130">Strict Versioning</span></span>  
 <span data-ttu-id="3a7cd-131">很多情况下，当更改版本是一个问题时，服务开发人员无法控制客户端，因此无法假设它们将对消息 XML 或架构中的更改作何反应。</span><span class="sxs-lookup"><span data-stu-id="3a7cd-131">In many scenarios when changing versions is an issue, the service developer does not have control over the clients and therefore cannot make assumptions about how they would react to changes in the message XML or schema.</span></span> <span data-ttu-id="3a7cd-132">在这些情况下，出于两个原因，必须保证将针对旧架构验证新消息：</span><span class="sxs-lookup"><span data-stu-id="3a7cd-132">In these cases, you must guarantee that the new messages will validate against the old schema, for two reasons:</span></span>  
  
-   <span data-ttu-id="3a7cd-133">开发旧客户端时的假设是架构将不会更改。</span><span class="sxs-lookup"><span data-stu-id="3a7cd-133">The old clients were developed with the assumption that the schema will not change.</span></span> <span data-ttu-id="3a7cd-134">这些旧客户端可能无法处理设计时并未考虑要处理的消息。</span><span class="sxs-lookup"><span data-stu-id="3a7cd-134">They may fail to process messages that they were never designed for.</span></span>

### <a name="lax-versioning"></a><span data-ttu-id="3a7cd-140">宽松版本管理</span><span class="sxs-lookup"><span data-stu-id="3a7cd-140">Lax Versioning</span></span>
