# <a name="peer-channel-security"></a><span data-ttu-id="a2347-102">对等通道安全性</span><span class="sxs-lookup"><span data-stu-id="a2347-102">Peer Channel Security</span></span>
<span data-ttu-id="a2347-103">对等通道启用依赖于多方消息传递的多种分布式应用程序类型。</span><span class="sxs-lookup"><span data-stu-id="a2347-103">Peer Channel enables a variety of distributed application types that depend on multiparty messaging.</span></span> <span data-ttu-id="a2347-104">示例包括在 Internet 范围内由受信任源分发内容（如媒体或软件更新）、朋友们一起交换音乐和图片或同事们一起协作编辑一个文档。</span><span class="sxs-lookup"><span data-stu-id="a2347-104">Some examples include Internet-scale content distribution, where a trusted source distributes content (such as media or software updates), a group of friends exchange music and photos, or a team of colleagues collaboratively edit a document.</span></span> <span data-ttu-id="a2347-105">这些方案都需要一个独特的安全模型。</span><span class="sxs-lookup"><span data-stu-id="a2347-105">Each of these scenarios requires a unique security model.</span></span> <span data-ttu-id="a2347-106">对等通道安全模型就是为满足这些方案而设计的，此模型为不同标识、身份验证和授权模型的各自需要提供可靠的安全模型。</span><span class="sxs-lookup"><span data-stu-id="a2347-106">The Peer Channel security model is designed to address these scenarios and provides a sound security model for the respective needs of different identity, authentication, and authorization models.</span></span>  
  
## <a name="security-scenarios"></a><span data-ttu-id="a2347-107">安全方案</span><span class="sxs-lookup"><span data-stu-id="a2347-107">Security Scenarios</span></span>  
 <span data-ttu-id="a2347-108">内容分发方案要求每个内容接收方都要标识内容源。</span><span class="sxs-lookup"><span data-stu-id="a2347-108">A content-distribution scenario requires that each content recipient identify the content source.</span></span> <span data-ttu-id="a2347-109">由于该方案的分布式性质，并不是总能够知道和信任处理或截取消息的中介。</span><span class="sxs-lookup"><span data-stu-id="a2347-109">Due to the distributed nature of the scenario, it is not always possible to know and trust the intermediaries that process or intercept messages.</span></span> <span data-ttu-id="a2347-110">为了有效缓解非受信任中介篡改消息的威胁，应用程序可以在发送方保护消息，以便能够容易地检测出任何篡改尝试。</span><span class="sxs-lookup"><span data-stu-id="a2347-110">To effectively mitigate the threat that an untrusted intermediary might tamper with the messages, applications can secure the message at the sender so that any tampering attempts are easily detected.</span></span> <span data-ttu-id="a2347-111">在这种情况下，根据内容的保密性，可能有必要进行加密。</span><span class="sxs-lookup"><span data-stu-id="a2347-111">In this case, depending on the confidentiality of the content, encryption can be necessary.</span></span>  
  
 <span data-ttu-id="a2347-112">像组文档协作这样的协作方案通常需要对参与会话的每个成员单独进行标识和身份验证。</span><span class="sxs-lookup"><span data-stu-id="a2347-112">Collaboration scenarios like group document collaboration often require that each member participating in the session be individually identified and authenticated.</span></span> <span data-ttu-id="a2347-113">这意味着必须有定义用户组和对这些组进行身份验证的机制才能进行安全的会话。</span><span class="sxs-lookup"><span data-stu-id="a2347-113">This means that a mechanism to define user groups and authenticate against those groups is necessary to have a secured session.</span></span> <span data-ttu-id="a2347-114">而且，应用程序可能需要在消息级别对每个消息进行身份验证来跟踪每个消息。</span><span class="sxs-lookup"><span data-stu-id="a2347-114">Moreover, applications might require tracing each message by authentication at the message level.</span></span> <span data-ttu-id="a2347-115">在这些类型的应用程序中，可以牺牲性能来实现更可靠的安全方案。</span><span class="sxs-lookup"><span data-stu-id="a2347-115">In these types of applications, performance can be sacrificed for a stronger security scheme.</span></span>  
  
 <span data-ttu-id="a2347-116">一组临时用户中进行的通信会话可能需要非正式的安全模型，如对该组中一般秘密的了解。</span><span class="sxs-lookup"><span data-stu-id="a2347-116">A communication session among a group of casual users can require informal security models, like knowledge of a common secret among the group.</span></span> <span data-ttu-id="a2347-117">对于这些类型的应用程序，具有便于建立和配置的安全模型比具有最强的身份验证形式或提供不可否认措施更重要。</span><span class="sxs-lookup"><span data-stu-id="a2347-117">For these types of applications, having a security model that is convenient to establish and configure is more important than having the strongest form of authentication or providing nonrepudiation measures.</span></span> <span data-ttu-id="a2347-118">对于这些方案，基于密码的身份验证机制有助于确保通信层的安全，同时仍允许进行消息身份验证。</span><span class="sxs-lookup"><span data-stu-id="a2347-118">For these scenarios, a password-based authentication mechanism helps to secure the communication layer while still allowing for message authentication.</span></span> <span data-ttu-id="a2347-119">基于密码的安全性是对等通道的默认设置。</span><span class="sxs-lookup"><span data-stu-id="a2347-119">Password-based security is the default setting for Peer Channel.</span></span>  
  
## <a name="token-types"></a><span data-ttu-id="a2347-120">标记类型</span><span class="sxs-lookup"><span data-stu-id="a2347-120">Token Types</span></span>  
 <span data-ttu-id="a2347-121">对等通道只能识别一种强标识令牌类型，即 X.509 证书，X.509 证书基于可实现的身份验证和授权类型，提供强标识模型。</span><span class="sxs-lookup"><span data-stu-id="a2347-121">Peer Channel recognizes a single token type for strong identification, X.509 certificates, which provide a strong identity model based on the type of authentication and authorization that can be implemented.</span></span> <span data-ttu-id="a2347-122">使用证书易于提供保密性和完整性。</span><span class="sxs-lookup"><span data-stu-id="a2347-122">Confidentiality and integrity are easily provided using certificates.</span></span> <span data-ttu-id="a2347-123">但 X.509 证书可能难以使用和部署。</span><span class="sxs-lookup"><span data-stu-id="a2347-123">However, X.509 certificates can be difficult to use and deploy.</span></span>  
  
 <span data-ttu-id="a2347-124">对等通道还可通过使用密码为简单应用程序提供支持。</span><span class="sxs-lookup"><span data-stu-id="a2347-124">Peer Channel also provides support for simple applications through the use of passwords.</span></span> <span data-ttu-id="a2347-125">应用程序可以根据提供的密码选择设置快速而简单的对等组。</span><span class="sxs-lookup"><span data-stu-id="a2347-125">Applications can choose to set up quick and simple peer groups based on a supplied password.</span></span> <span data-ttu-id="a2347-126">在这种情况下，组所有者确定密码并将密码通知成员。</span><span class="sxs-lookup"><span data-stu-id="a2347-126">In this case, a group owner decides and communicates the password to members.</span></span> <span data-ttu-id="a2347-127">每个成员必须先使用此密码登录，才能加入会话。</span><span class="sxs-lookup"><span data-stu-id="a2347-127">Each member must sign in using this password before they can join the session.</span></span> <span data-ttu-id="a2347-128">密码仅用于允许加入会话，不能用于执行消息身份验证。</span><span class="sxs-lookup"><span data-stu-id="a2347-128">Passwords can be used only to allow entry to the session; they cannot be used to perform message authentication.</span></span> <span data-ttu-id="a2347-129">这是因为对等端组共享的对称令牌难以、也不适合用于进行源身份验证。</span><span class="sxs-lookup"><span data-stu-id="a2347-129">This is because a symmetric token that a group of peers share is difficult and inappropriate to use for source authentication.</span></span>  
  
## <a name="security-model"></a><span data-ttu-id="a2347-130">安全模型</span><span class="sxs-lookup"><span data-stu-id="a2347-130">Security Model</span></span>  
 <span data-ttu-id="a2347-131">对等通道提供保证对等端之间各个链接安全的功能。</span><span class="sxs-lookup"><span data-stu-id="a2347-131">Peer Channel provides the ability to secure the individual links between peers.</span></span> <span data-ttu-id="a2347-132">这意味着消息决不会在不安全的链接上流动（从应用程序角度讲）。</span><span class="sxs-lookup"><span data-stu-id="a2347-132">This means that a message never flows on an unsecured link (from the application perspective).</span></span> <span data-ttu-id="a2347-133">在内部，每个链接（两个对等端之间的传输通道）都使用传输层安全 (TLS) 保证安全。</span><span class="sxs-lookup"><span data-stu-id="a2347-133">Internally, each link (a transport channel between two peers) is secured using Transport Layer Security (TLS).</span></span> <span data-ttu-id="a2347-134">这意味着某一发送方在撰写并发送消息时，消息会通过安全传输发送给该发送方的每个中间对等端，这些对等端访问消息，然后通过安全连接将消息发送到他们的中间对等端。</span><span class="sxs-lookup"><span data-stu-id="a2347-134">This means that when a sender composes and sends a message, it is sent over secure transport to each of its immediate peers, who access the message, and in turn send the message to their immediate peers over secure connections.</span></span> <span data-ttu-id="a2347-135">此安全模型仅在传输级别有效并独立于消息安全模型。</span><span class="sxs-lookup"><span data-stu-id="a2347-135">This security only works at the transport level and is independent of the message security models.</span></span>  
  
 <span data-ttu-id="a2347-136">对等通道还提供一种保护消息独立于所使用的传输安全的途径。</span><span class="sxs-lookup"><span data-stu-id="a2347-136">Peer Channel also provides a way to secure messages independently of the transport security used.</span></span> <span data-ttu-id="a2347-137">在此模型中，虽然当前只支持 X.509 证书，但消息在源上可以使用源的安全令牌加以保护。</span><span class="sxs-lookup"><span data-stu-id="a2347-137">In this model, the message is secured at the source using the source’s security token, although currently only X.509 certificates are supported.</span></span> <span data-ttu-id="a2347-138">然后通过对等网络传输安全消息。</span><span class="sxs-lookup"><span data-stu-id="a2347-138">The secured message is then transmitted over the peer network.</span></span> <span data-ttu-id="a2347-139">每个接收对等端都可以验证源的真实性。</span><span class="sxs-lookup"><span data-stu-id="a2347-139">Each receiving peer can verify the authenticity of the source.</span></span> <span data-ttu-id="a2347-140">请注意，消息具有安全保护，中介无法篡改。</span><span class="sxs-lookup"><span data-stu-id="a2347-140">Note that the message is secured so that intermediaries cannot tamper with it.</span></span>  
  
 <span data-ttu-id="a2347-141">为了实现保密性，应用程序可使用具有可靠组成员资格方案的传输安全以防止未经授权访问该消息。</span><span class="sxs-lookup"><span data-stu-id="a2347-141">To achieve confidentiality, applications can employ transport security with strong group membership schemes to prevent unauthorized access to the message.</span></span>  
  
 <span data-ttu-id="a2347-142">只要应用程序选择了一种受支持的令牌类型，对等通道就不再需要特定的标识模型。</span><span class="sxs-lookup"><span data-stu-id="a2347-142">Peer Channel does not require a specific identity model as long as the application chooses one of the supported token types.</span></span> <span data-ttu-id="a2347-143">应用程序可以完全控制这些标识和身份验证决策的生命周期。</span><span class="sxs-lookup"><span data-stu-id="a2347-143">Applications completely own the life cycle of these identities and authentication decisions.</span></span>
