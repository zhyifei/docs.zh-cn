## <a name="basic-ssl"></a><span data-ttu-id="d9ac8-106">基本 SSL</span><span class="sxs-lookup"><span data-stu-id="d9ac8-106">Basic SSL</span></span>  
 <span data-ttu-id="d9ac8-107">SSL 的工作方式可以通过一个典型方案得到最好的说明，在本示例中，该方案为银行的网站。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-107">How SSL works is best explained through a typical scenario, in this case, a bank's Web site.</span></span> <span data-ttu-id="d9ac8-108">该网站允许客户使用用户名和密码登录。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-108">The site allows a customer to log on with a user name and password.</span></span> <span data-ttu-id="d9ac8-109">在经过身份验证之后，用户可以执行事务，例如查看帐户余额、支付帐单以及将钱从一个帐户转到其他帐户。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-109">After being authenticated, the user can perform transactions, such as view account balances, pay bills, and move money from one account to another.</span></span>  
  
 <span data-ttu-id="d9ac8-110">当用户首次访问站点时，SSL 机制启动一系列的协商，称为*握手*，与用户的客户端 （在此情况下，Internet Explorer）。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-110">When a user first visits the site, the 首先向客户证明银行网站的身份。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-111">SSL first authenticates the bank site to the customer.</span></span> <span data-ttu-id="d9ac8-112">这一步骤是必需的，因为客户首先必须知道他们正在与真实网站进行通信，而不是与一个试图引诱他们键入自己的用户名和密码的诈骗网站进行通信。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-112">This is an essential step because customers must first know that they are communicating with the actual site, and not a spoof that tries to lure them into typing in their user name and password.</span></span> <span data-ttu-id="d9ac8-113">SSL 通过使用由受信任的颁发机构（例如 VeriSign）提供的 SSL 证书来执行此身份验证。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-113">SSL does this authentication by using an SSL certificate provided by a trusted authority, such as VeriSign.</span></span> <span data-ttu-id="d9ac8-114">其逻辑如下：VeriSign 担保该银行网站的身份是真实的。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-114">The logic goes like this: VeriSign vouches for the identity of the bank site.</span></span> <span data-ttu-id="d9ac8-115">由于 Internet Explorer 信任 VeriSign，因此也信任该网站。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-115">Because Internet Explorer trusts VeriSign, the site is trusted.</span></span> <span data-ttu-id="d9ac8-116">如果您希望向 VeriSign 进行核实，可以通过单击 VeriSign 徽标执行此操作。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-116">If you want to check with VeriSign, you can do so as well by clicking on the VeriSign logo.</span></span> <span data-ttu-id="d9ac8-117">这将显示一份含有到期日期以及接受方（银行网站）的真实性声明。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-117">That presents a statement of authenticity with its expiration date and who it is issued to (the bank site).</span></span>  
  
 <span data-ttu-id="d9ac8-118">若要启动一个安全会话，客户端向服务器发送一个等效于“你好”的项，连同一个它可以用来签名、生成哈希以及进行加密和解密的加密算法列表。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-118">To initiate a secure session, the client sends the equivalent of a "hello" to the server along with a list of cryptographic algorithms it can use to sign, generate hashes, and encrypt and decrypt with.</span></span> <span data-ttu-id="d9ac8-119">作为响应，该网站发送回一个确认以及它对算法组之一的选择。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-119">In response, the site sends back an acknowledgment and its choice of one of the algorithms suites.</span></span> <span data-ttu-id="d9ac8-120">在该初次握手期间，双方都发送和接收 Nonce。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-120">During this initial handshake, both parties send and receive nonces.</span></span> <span data-ttu-id="d9ac8-121">一个*nonce*是一个随机生成的是，与结合使用站点的公钥，若要创建哈希的数据。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-121">A *nonce* is a randomly generated piece of data that is used, in combination with the site's public key, to create a hash.</span></span> <span data-ttu-id="d9ac8-122">一个*哈希*是从两个数字使用标准算法，如 SHA1 派生一个新数字。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-122">A *hash* is a new number that is derived from the two numbers using a standard algorithm, such as SHA1.</span></span> <span data-ttu-id="d9ac8-123">（客户端和网站还交换消息以协商要使用的哈希算法。）哈希是唯一的，并且仅在客户端和网站之间的会话中使用以便对消息进行加密和解密。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-123">(The client and the site also exchange messages to agree which hash algorithm to use.) The hash is unique and is used only for the session between the client and the site to encrypt and decrypt messages.</span></span> <span data-ttu-id="d9ac8-124">客户端和服务都具有原始 Nonce 和证书的公钥，所以通信两端可以生成同一个哈希。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-124">Both client and service have the original nonce and the certificate's public key, so both sides can generate the same hash.</span></span> <span data-ttu-id="d9ac8-125">因此，客户端可以通过以下方式验证服务所发送的哈希：(a) 使用商定的算法根据数据计算哈希；并 (b) 将计算出的哈希与服务所发送的哈希进行比较。如果二者匹配，则客户端可以确信该哈希未遭篡改。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-125">Therefore, the client validates the hash sent by the service by (a) using the agreed upon algorithm to calculate the hash from the data, and (b) comparing it to the hash sent by the service; if the two match, then the client has assurance that the hash has not been tampered with.</span></span> <span data-ttu-id="d9ac8-126">客户端随后可以将此哈希用作密钥，以便对同时包含另一个新哈希的消息进行加密。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-126">The client can then use this hash as a key to encrypt a message that contains yet another new hash.</span></span> <span data-ttu-id="d9ac8-127">服务可以使用此哈希对消息进行解密，并重新获得倒数第二个哈希。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-127">The service can decrypt the message using the hash, and recover this second-to-final hash.</span></span> <span data-ttu-id="d9ac8-128">这样，通信双方就都获知累积的信息（Nonce、公钥和其他数据），并且可以创建最后一个哈希（也称主密钥）。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-128">The accumulated information (nonces, public key, and other data) is now known to both sides, and a final hash (or master key) can be created.</span></span> <span data-ttu-id="d9ac8-129">这个最终的密钥使用倒数第二个哈希加密后发送。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-129">This final key is sent encrypted using the next-to-last hash.</span></span> <span data-ttu-id="d9ac8-130">然后，使用主密钥对用来重置会话的消息进行加密和解密。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-130">The master key is then used to encrypt and decrypt messages for the reset of the session.</span></span> <span data-ttu-id="d9ac8-131">由于客户端和服务使用相同的密钥，这也称为*会话密钥*。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-131">Because both client and service use the same key, this is also called a *session key*.</span></span>  
  
 <span data-ttu-id="d9ac8-132">会话密钥还被描述为对称密钥，或“共享秘密”。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-132">The session key is also characterized as a symmetric key, or a "shared secret."</span></span> <span data-ttu-id="d9ac8-133">具有对称密钥很重要，因为它减少了事务双方所需执行的计算量。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-133">Having a symmetric key is important because it reduces the computation required by both sides of the transaction.</span></span> <span data-ttu-id="d9ac8-134">如果每个消息都要求对 Nonce 和哈希进行新的交换，那么性能将会下降。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-134">If every message demanded a new exchange of nonces and hashes, performance would deteriorate.</span></span> <span data-ttu-id="d9ac8-135">因此，SSL 的最终目标是使用允许消息在通信双方之间自由流动的对称密钥，同时具有更高程度的安全和效率。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-135">Therefore, the ultimate goal of SSL is to use a symmetric key that allows messages to flow freely between the two sides with a greater degree of security and efficiency.</span></span>  
  
 <span data-ttu-id="d9ac8-136">因为协议可能因网站而异，所以前面的描述只是所发生过程的简化版本。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-136">The previous description is a simplified version of what happens, because the protocol may vary from site to site.</span></span> <span data-ttu-id="d9ac8-137">还有一种可能，就是客户端和网站都在握手期间生成在算法上相结合的 Nonce，以增加数据交换过程的复杂性，从而为该过程提供更多的保护。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-137">It is also possible that both the client and the site both generate nonces that are algorithmically combined during the handshake to add more complexity, and therefore protection, to the process of exchanging data.</span></span>  
  
### <a name="certificates-and-public-key-infrastructure"></a><span data-ttu-id="d9ac8-138">证书和公钥基础结构</span><span class="sxs-lookup"><span data-stu-id="d9ac8-138">Certificates and Public Key Infrastructure</span></span>  
 <span data-ttu-id="d9ac8-139">在握手期间，服务还将其 SSL 证书发送到客户端。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-139">During the handshake, the service also sends its SSL certificate to the client.</span></span> <span data-ttu-id="d9ac8-140">该证书包含一些信息，例如证书的到期日期、颁发机构以及网站的统一资源标识符 (URI)。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-140">The certificate contains information, such as its expiration date, issuing authority, and the site's Uniform Resource Identifier (URI).</span></span> <span data-ttu-id="d9ac8-141">客户端将该 URI 与它原来联系的 URI 进行比较，以确保二者匹配，并且对日期和颁发机构进行检查。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-141">The client compares the URI to the URI it had originally contacted to ensure a match, and also checks the date and issuing authority.</span></span>  
  
 <span data-ttu-id="d9ac8-142">每个证书具有两个密钥、 私钥和公钥，并且这两个名为*交换密钥对*。</span><span class="sxs-lookup"><span data-stu-id="d9ac8-142">Every certificate has two keys, a private key and a public key, and the class="sxs-lookup"><span data-stu-id="d9ac8-147">This provides assurance to the client that the messages are being exchanged only with the possessor of the private key because only messages encrypted site is assured that it is exchanging messages with a client that has encrypted using the public key.</span></span> <span data-ttu-id="d9ac8-149">然而，这种交换仅对初次握手是安全的，因此在创建实际的对称密钥时要复杂得多。</span><span
